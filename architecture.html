Section 01

What It Does

Cephlow is a certificate generation and delivery SaaS. Organizations use it to bulk-generate and send PDF certificates to students via email and WhatsApp. A user connects their Google account, picks a Google Sheet (participant list) and a Google Slides template (certificate design), then clicks Generate. The app fills the template for each row, exports it as a PDF, uploads it to Cloudflare R2, and sends it via Gmail or WhatsApp Business API.

Recipients can text the WhatsApp number at any time to re-download their certificate. A Cloudflare Worker handles the bot and bridges customer support conversations to Telegram topics.

Key Pattern — OpenAPI Codegen

api-spec → codegen → api-zod + api-client-react. The OpenAPI spec is the single source of truth for types. You change the spec, run codegen, and both server-side Zod validation and client-side React Query hooks stay in sync automatically. This eliminates manual type duplication between frontend and backend.

Section 06

Auth & Database — Supabase (Free Tier)

Supabase provides PostgreSQL + Auth + Row Level Security + PostgREST API. It sits on the free tier (500MB DB, 50K MAU). The supabaseAdmin client uses the service role key — it bypasses RLS and is backend-only.

Key Tables

TablePurposeKey Columns
batches One row per generation job user_id, template_id, sheet_id, status, total_count, generated_count, sent_count
certificates One row per generated cert batch_id, recipient_name, r2_pdf_url, whatsapp_status, wamid, sent_at
tasks Background job queue (poll-based) type (send_email/send_whatsapp), payload (JSON), status, error_message
wa_messages Maps WhatsApp wamid → cert wamid, batch_id, cert_id — used for delivery receipt updates
wa_tg_threads WhatsApp ↔ Telegram topic mapping phone_e164, telegram_topic_id, supergroup_id
user_profiles Google OAuth + approval status user_id, google_refresh_token, is_approved, workspace_id
wallet_ledger Credit debit/credit history user_id, amount, type, balance_after, metadata
orders Cashfree payment orders order_id, user_id, amount, status — idempotency key for webhooks
◆
Why Postgres as a Task Queue?

The project originally used BullMQ with Redis for background jobs. Redis was removed to cut costs. A Supabase RPC grab_pending_tasks() does an atomic UPDATE ... RETURNING * to claim tasks — no two workers can grab the same task. This is a well-known pattern called "Postgres as a poor man's queue." Slightly higher latency than Redis (poll every 2s vs. event push) but zero additional infrastructure cost.

Section 07

Certificate Generation Pipeline

▶
Key Design: Client-Side Generation

The browser generates all PDFs — the server is not involved in rendering. This was migrated from server-side in commit abfec93 to fix OOM crashes on the $18 droplet for large batches, and to use each user's personal Google API quota instead of the shared project quota.

01
User Configures Batch
Selects Google Sheet (participant data) + Google Slides template + column mapping (which sheet column = name, email, phone, etc).
02
Frontend Fetches Data
GET /api/sheets/:sheetId/data returns all rows. GET /api/auth/google/access-token returns a short-lived Google access token the browser will use directly.
03
Browser: Copy Template
For each row, browser calls Google Drive API to copy the Slides template file (creating a new file per recipient).
04
Browser: Fill Placeholders
Calls Google Slides API batchUpdate to replace {{Name}}, {{Event}}, etc. with the row's actual values. Font scaling heuristics ensure text fits.
05
Browser: Export to PDF
Calls Google Drive export endpoint: GET /drive/v3/files/:id/export?mimeType=application/pdf. Returns the PDF bytes in the browser.
06
Get Presigned R2 URL
Browser calls POST /api/batches/:id/client-generate to get a presigned S3-compatible PUT URL for Cloudflare R2. This is the only server call per certificate.
07
Browser: Direct Upload to R2
Browser does PUT presignedUrl with the PDF bytes. The file goes directly from browser → Cloudflare R2. The API server never sees the PDF bytes at all.
08
Record in DB
Browser calls POST /api/batches/:id/client-generate/complete with the R2 URL. Server inserts the certificate record into Supabase with status = "generated".
09
Send (Optional)
User clicks "Send Email" or "Send WhatsApp." Server enqueues send_email or send_whatsapp tasks in Supabase. The background worker picks them up and delivers.

R2 Storage Path Convention

R2 Object Key Format
{phone_without_country_code}/{recipient_name_slug}.pdf

Example:
  9876543210/Adithyan-B-Raj.pdf      ← Indian number without +91
  919876543210/Adithyan-B-Raj.pdf    ← E.164 format with country code

The WhatsApp bot tries BOTH variants when listing a user's certs.

◆
Template Pre-Approval

WhatsApp Business templates must be approved by Meta before use. document_senderv2 was submitted and approved — it allows sending a document as the header, with 3 text body variables and a URL button. You cannot send free-form messages to users who haven't messaged you first (24-hour window rule). Templates bypass this restriction.

Delivery Status Tracking

01
Send → wamid Stored
WhatsApp API returns a wamid (message ID). Server stores it in wa_messages table linked to the certificate.
02
Meta Fires Status Webhooks
Meta sends sent → delivered → read webhooks to the Cloudflare Worker endpoint.
03
Worker Forwards to API
If the webhook contains statuses (not a message), the Worker forwards the payload to POST /api/webhooks/whatsapp on the Express server.
04
Server Updates DB
Looks up wamid in wa_messages → finds the cert_id → updates certificates.whatsapp_status to delivered or read.

Why Telegram Supergroup Topics?

Each customer gets their own "topic" (thread) in a single Telegram supergroup. The developer sees all support conversations organized per customer, with full media support. It's a free, zero-infrastructure customer support desk. The wa_tg_threads table in Supabase acts as the routing table: phone ↔ telegram_topic_id.

Section 11

Background Worker (Email + WhatsApp Delivery)

A separate Node.js process (worker.ts) runs alongside the Express server on the same droplet. It polls Supabase every 2 seconds for pending tasks and executes them concurrently.

apps/api-server/src/worker.ts — Poll Loop
every 2 seconds:
  1. supabaseAdmin.rpc("grab_pending_tasks", { p_limit: 10 })
     → Postgres RPC: UPDATE tasks SET status='processing'
       WHERE status='pending' LIMIT 10 RETURNING *
     → Atomic claim — no two workers can grab the same task

  2. Promise.all(tasks.map(executeTask))
     → Run all 10 tasks concurrently

  3. Per task:
     if type === "send_email"    → processSendEmail(payload)
     if type === "send_whatsapp" → processSendWhatsApp(payload)

  4. Mark status = "completed" or "failed" (with error_message)
send_email task
Gmail API
  • Uses organizer's own Gmail (OAuth2)
  • Sends from their personal email address
  • PDF link in email body (R2 URL)
send_whatsapp task
WA Business API
  • Uses pre-approved template document_senderv2
  • Returns wamid → stored in wa_messages
  • Delivery tracking via Meta webhooks

CORS Strategy for Google Fonts

Google Fonts CSS is fetched via the API server as a proxy to avoid CORS issues in the browser during client-side rendering. Font files are cached in memory. The graph report marks this as "Google API CORS Strategy" — a non-obvious architectural detail interviewers might ask about.

Section 13

Payments — Cashfree + Wallet Credits

Uses Cashfree (Indian payment gateway) with a wallet-credit model. Users top up their wallet, credits are deducted per certificate generated.

01
Create Order
POST /api/payments/order → creates a Cashfree order and returns an order token to the frontend.
02
Checkout
Cashfree SDK rendered in browser handles UPI / card / netbanking. No payment data ever hits our server.
03
Webhook Fires
Cashfree sends PAYMENT_SUCCESS_WEBHOOK to POST /api/webhooks/cashfree. Server verifies the HMAC signature on the raw request body before processing.
04
Atomic Credit (Idempotent)
Supabase RPC process_payment() checks if the order_id was already processed. Returns already_processed if so. Otherwise inserts ledger entry atomically. Prevents double-crediting from duplicate webhooks.
Section 14

Key Design Decisions

These are the architectural choices most likely to come up in interviews. Each has a clear "why."

DecisionWhatWhy
Client-side generation Browser renders and exports PDFs Server OOM on $18 droplet for large batches. Also uses user's own Google API quota.
Presigned R2 URLs Browser uploads directly to R2 Server never handles large payloads. Saves bandwidth and CPU on the droplet.
Postgres as task queue Supabase tasks table + poll worker Replaced BullMQ/Redis after Redis cost became a problem. Atomic UPDATE claim prevents duplicate processing.
Cloudflare Worker for WA bot Edge function, not on the droplet Zero infrastructure, global low latency, free D1 SQLite, isolates bot from main server load.
WhatsApp ↔ Telegram bridge Worker bridges messages to TG topics Free customer support desk. Developer can reply from Telegram; customer sees it on WhatsApp.
OpenAPI codegen Spec → Zod + React Query hooks Single source of truth for types. No manual duplication between frontend and backend.
JWKS + HMAC fallback JWT verification with two strategies Supabase changed key format (HMAC → RS256). Fallback ensures zero-downtime migration.
Per-user Google OAuth Each organizer uses their own Google account Gmail sends from their own email (trusted). Slides edits are in their Drive. Per-user API quotas.
Section 15

Interview Q&A

Click any question to expand the answer.

Explain your system design in 60 seconds. ▶
It's a multi-tenant certificate SaaS on a $18 DigitalOcean droplet running Express. Certificate generation is client-side — the browser connects to Google APIs with the user's OAuth token, fills in the template, exports a PDF, and uploads it directly to Cloudflare R2 via a presigned URL. The server records the result in Supabase. For delivery, a background worker on the same droplet polls a Supabase tasks table every 2 seconds and executes email and WhatsApp sends. The WhatsApp bot is a separate Cloudflare Worker edge function with its own D1 SQLite for analytics. A Telegram bridge lets our developer support customers from Telegram while they see it on WhatsApp.
Why did you move from server-side to client-side certificate generation? ▶
The server was crashing with out-of-memory errors on large batches — generating 200 PDFs concurrently was too much for the $18 droplet. Moving to client-side uses the browser's memory and CPU. It also moves the Google API calls into the user's own OAuth context, so we use their per-user quota instead of a shared project quota. The presigned R2 URL pattern means the server never even sees the PDF bytes — the browser uploads directly to object storage.
How does the WhatsApp ↔ Telegram support bridge work? ▶
When a student taps "Talk to Developer," the Cloudflare Worker sets their state to support_active in D1 and creates a forum topic in a Telegram supergroup using the createForumTopic API. The topic ID is stored in a wa_tg_threads table in Supabase keyed by phone number. Every subsequent WhatsApp message from that student — text or media — is forwarded to that Telegram topic. For media, the Worker downloads the file from the WhatsApp Media API and re-uploads it to Telegram. When the developer replies in the Telegram topic, Telegram fires a webhook to /telegram on the Worker. The Worker reverse-looks up the thread ID in wa_tg_threads to find the phone number, downloads the media from Telegram if needed, uploads it to the WhatsApp Media API to get a media_id, and sends it to the customer. The student sees a seamless conversation.
How do you prevent duplicate task execution in the background worker? ▶
The Supabase RPC grab_pending_tasks() runs a single SQL statement: UPDATE tasks SET status='processing' WHERE id IN (SELECT id FROM tasks WHERE status='pending' LIMIT 10 FOR UPDATE SKIP LOCKED) RETURNING *. The FOR UPDATE SKIP LOCKED is key — it atomically locks and claims rows. Two concurrent worker instances (or two poll intervals) cannot grab the same task because the lock is held during the UPDATE. This is the standard Postgres queue pattern.
How do you handle payment idempotency? ▶
Cashfree webhooks can fire multiple times for the same payment (retries on timeout). The process_payment Supabase RPC checks if the order_id already exists in the wallet_ledger before inserting. If it does, it returns already_processed without crediting again. This entire check-and-insert runs inside a single SQL transaction, so there's no race condition even with concurrent webhook deliveries.
Why use Cloudflare Worker instead of Express routes for the WhatsApp bot? ▶
Three reasons. First, isolation — the bot runs independently of the API server, so a bot spike doesn't affect the main app. Second, cost — Cloudflare Worker free tier handles millions of requests and includes D1 SQLite for analytics, all free. Third, performance — Workers run at the edge globally, so WhatsApp webhook processing has low latency regardless of where Meta's servers deliver the webhook from.
Explain the monorepo's codegen strategy. ▶
The api-spec package holds a single OpenAPI YAML spec that defines every endpoint, request body, and response shape. A codegen script reads this and generates two things: api-zod (Zod schemas for server-side runtime validation) and api-client-react (React Query hooks for the frontend). When you add a new endpoint, you update the spec once, run codegen, and both the server validation and the client fetching code update automatically. There's no manual type duplication between frontend and backend.
What happens if the DigitalOcean droplet runs out of memory? ▶
The main risk is during large batch generation — but that's now client-side, so the server isn't involved. The biggest server-side memory consumers are the background worker (10 concurrent tasks, mostly I/O-bound) and Express itself. The WhatsApp bot is on a separate Cloudflare Worker, so it's unaffected. For the future, the natural scale path is to move the background worker to a separate process or a managed queue service like Inngest or Trigger.dev.
