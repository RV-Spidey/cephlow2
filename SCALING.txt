CURRENT ARCHITECTURE BOTTLENECKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. EXPRESS SERVER HOLDING LONG HTTP CONNECTIONS ❌
   - When a user clicks "Generate", the HTTP request stays open
     until ALL certificates are done (can take 25+ minutes for 500 certs)
   - 5000 users generating simultaneously = 5000 open connections
   - Server runs out of memory and crashes
   - This is the ONLY real bottleneck

2. GOOGLE API QUOTA ✅ NOT A PROBLEM
   - Every API call (Slides, Drive, Sheets) uses the USER's OAuth token
   - Each user has their own independent quota (60 writes/min)
   - 5000 users generating = 5000 separate Google quotas
   - Your Google Cloud project quota is never touched during generation
   - Per user: ~20 certs/minute (limited by 60 writes/min quota)
   - 500 cert batch = ~25 minutes per user

3. SUPABASE CONNECTIONS ✅ NOT A PROBLEM
   - You use the Supabase JS client which talks over HTTP (PostgREST)
   - Does NOT use raw PostgreSQL TCP connections
   - Connection pool limits don't apply to the JS client
   - Would only matter if you switched to pg/prisma direct connections

4. CLOUDFLARE R2 ✅ NOT A PROBLEM
   - Scales infinitely, zero egress fees
   - No rate limits that affect certificate generation

How it works now (the problem):


User clicks Generate
       ↓
API server starts generating cert 1... cert 2... cert 3... (25 mins)
       ↓
Response sent after ALL done
User waits 25 mins, connection stays open the whole time.

How job queue works:


User clicks Generate
       ↓
API adds job to queue → instantly responds "generation started"
       ↓
Separate worker process picks up the job
       ↓
Worker generates cert 1... cert 2... cert 3... (in background)
       ↓
Frontend polls "how many done?" every 3 seconds → shows progress bar
Simple analogy:

Think of a restaurant. Currently your waiter (server) goes to the kitchen, cooks the food himself, then comes back — blocking all other customers. With a job queue, the waiter takes the order, passes it to the kitchen (worker), and immediately goes to serve the next customer. Kitchen works in the background.

The tech:

BullMQ — the queue manager (holds jobs, retries failures)
Redis — where the queue is stored (BullMQ needs Redis)
Worker — a separate Node.js process that reads from the queue and does the actual generation
What changes in your code:

POST /batches/:id/generate — instead of generating, just adds a job:


await generationQueue.add('generate', { batchId, userId })
res.json({ status: 'started' })
New worker.ts file — does the actual generation:


worker.process('generate', async (job) => {
  const { batchId, userId } = job.data
  // all the current generation code goes here
})
